# Wolves Storage Sussex — Build Rules (SEO · Design · EEAT)

The standard every page on this site must follow. Mirrors the wolves-removals
design system + SEO/EEAT bible. **On any new page build, follow these rules.**

---

## 1. Design rules (brand tokens — from the wolves-removals theme)

- **Font:** Barlow, self-hosted woff2 in `/fonts` (Regular 400 / Medium 500 / Semibold 600 / Bold 700). No external Google Fonts.
- **Colours:**
  - Body text `#262626`
  - Top contact bar (darkgrey) `#697783`
  - Header / menu bar (beige) `#E8E6DA`
  - Section backgrounds alternate: white `#FFFFFF` · lightgrey `#F9F8F6` · beige `#E8E6DA` · dark `navy-800 #2c2825`
  - **Primary CTA / accent (orange) `#FC9700`** — white, uppercase, tracking-wider text
  - Review stars (gold) `#F6BB06`
  - Borders `#E7E7E7`
- **Header:** darkgrey contact strip (two phones + email) above a beige menu bar; oversized circular logo absolutely positioned, **overlapping onto the hero** with `drop-shadow`; nav uppercase Barlow-bold, hover/active orange; orange `GET A FREE QUOTE` pill; full nav at `xl`, hamburger below. Nav bar height `h-24 md:h-[116px] lg:h-[140px]` so the logo never fouls the text. **No visible breadcrumb bar** (BreadcrumbList schema only).
- **Hero:** `min-h-[30rem] lg:min-h-[36rem] flex items-center`, real WebP photo + dark `hero-overlay`; eyebrow → H1 → subhead → white reviews badge (★ `#F6BB06` + "478 reviews · Trustindex") → green-check benefit list → orange CTA.
- **Sections:** `py-12 lg:py-20`, `max-w-6xl` container, `border-b border-[#E7E7E7]`, alternating bg; eyebrow (orange uppercase) + H2 (Barlow extrabold navy) + lead; cards `bg-white rounded-2xl ring-1 ring-[#E7E7E7]`.
- **Buttons:** orange pill (primary), navy pill (secondary), ghost-on-dark.
- **Interactivity:** reuse shared `script.js` — mobile menu, quote modal, testimonial carousel, FAQ accordion, image lightbox, fake forms (success message), reveal-on-scroll, counters, cookie banner, back-to-top.

## 2. Image rules

- **Every photo is WebP.** SVG only for icons / logo / placeholders (vector).
- **Every image has descriptive, keyword-relevant `alt` text.**
- Hero image: `fetchpriority="high"` (eager). All other images `loading="lazy" decoding="async"`.
- Every `<img>` has a `data-fallback` to a local branded SVG placeholder (offline-safe).
- Use the company's **real photos**; use a branded placeholder only where no real photo exists (swap later).

## 3. SEO rules (every page must pass)

- Unique `<title>` ≤ 60 chars (keyword + West Sussex / town + brand).
- Unique meta description 140–160 chars with a call to action.
- Exactly one `<h1>`; logical H2/H3 hierarchy.
- `canonical`, `robots`, Open Graph (type/title/url/image), `og:locale en_GB`, Twitter card (title/desc/image).
- **Schema:** global LocalBusiness / SelfStorage on every page (NAP, geo, opening hours, `aggregateRating` + real reviews); plus page-specific `Service`, `BreadcrumbList`, and `FAQPage` (matching the on-page FAQ).
- Keyword-rich filenames/URLs; descriptive internal links (never "click here"); several internal links per page.
- geo meta (`geo.region` GB-WSX / `geo.placename` / `geo.position` / `ICBM`); `robots.txt` references the sitemap; favicon + apple-touch + `site.webmanifest`.
- **RULE — every new page is auto-listed in `sitemap.xml`.** The sitemap is generated by `build-site.py` from the page list `P`; never hand-edit it. Build a page → it's in the sitemap.
- **RULE — every new page is auto-listed in `llms.txt`.** `build-site.py` also writes a root `llms.txt` (llmstxt.org format: H1 + `>` summary + grouped `- [Title](url): description` links) from the same page list, so the LLM-readable index never drifts. Build a page → it's in `llms.txt`.
- Mobile-responsive; fast (self-hosted fonts, WebP, lazy images, `script.js` at end of body).
- `404.html` is `noindex`.

## 5. Page-build process

- **Storage pages:** extend `build-storage-pages.py` — shared head/header/footer partials + 21 section templates + per-page data dict — so the chrome stays identical and these rules are enforced by construction. Run it to (re)generate.
- **Other pages:** clone head/header/footer from an existing page; swap title/meta/canonical/OG/Twitter + page schema; build `<main>`; then run the audit (single H1, valid JSON-LD, 100% WebP + alt, no broken links, meta present).

## 5a. Service-page template (RULE — all service/"money" pages)

**Every service page must follow the `long-term-storage.html` design**, in this section order:
1. Hero (eyebrow → H1 → subhead → checklist → CTA)
2. Intro content split (`split`, bg-white)
3. **On-page pricing + "what fits" panel** (`town_pricing` — subject-headed, e.g. "Furniture Storage Prices in West Sussex")
4. Second content split (`split`, bg-lightgrey, reversed)
5. **Subject use-cases** ("When … Helps", `svc_situations` — grey panel, 4 cards)
6. **"Why West Sussex Chooses Wolves for …"** differentiators (`svc_whychoose` — dark glass band, 4 points)
7. Step-by-step **process** carousel
8. **Gallery** (`gallery` — branded tiles + lightbox)
9. **8-question FAQ** (`faq`, schema-matched)
10. Container-vs-self-storage **explainer** (`CONTAINER_DATA` entry — dark band, image + ticks)
11. **Location silo** ("Local Storage Across Sussex", `all_towns_strip` — grouped directory)
12. CTA band

Enforced by construction: the four pages built by `service()` (storage-solutions, long-term, short-term, business) are in `EXPAND_SERVICES`; `furniture-storage` (standalone `furniture_page`) is aligned manually and must stay in step. Any new service page goes through `service()` (add it to `EXPAND_SERVICES` + `CONTAINER_DATA`).

**SEO RULE — same design, UNIQUE content.** The structure is identical across service pages, but **every use-case, "why-choose" point, FAQ, heading, lead and CTA must be written specifically for that service** (e.g. business → stock/archives/tools/relocation; short-term → chain delays/staging/renovations/tenancies). Never reuse boilerplate across service pages — identical structure + identical copy = near-duplicate pages that dilute rankings. Each page also carries `Service` + `FAQPage` + `SelfStorage` schema and targets ~1,200–1,800 useful words.

## 6. Toolchain notes

- WebP encoding via Python **Pillow** (`from PIL import Image; im.save(out, "WEBP", quality=80, method=6)`) — macOS `sips` can read but **not write** WebP.
- Local preview: `python3 -m http.server 8000` from the site root → http://localhost:8000.
