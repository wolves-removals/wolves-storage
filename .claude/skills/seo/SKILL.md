```
edit build-site.py  →  python3 build-site.py  →  python3 .claude/skills/seo/seo_scan.py

```
"Done" means: build succeeds (43+ pages), **0 hard SEO issues**, no new real bug-scan findings,
word floors held, and the render checked. Never claim PASS without running the scanners.

## 1. THE BIBLE — per-page rules R1–R19 (hard-fail unless marked *warn*)
(There is no R18 — the bible skips it. Canonical: BUILD-RULES.md + [[wolves-storage-seo-rules]].)

- **R1** — `<meta charset>`, `<meta viewport>`, `<html lang>` all present
- **R2** — `<title>` present, unique site-wide; ≤65 chars (*warn* >60)
- **R3** — meta description present, unique, **≤145 chars** (scanner: *warn* 146–160, **hard-fail >160**)
- **R4** — exactly one `<h1>`, unique site-wide
- **R5** — self-referencing canonical matching the page's real URL (`https://www.sussexstoragecompany.co.uk/<page>`)
- **R6** — OG + Twitter tags; `og:image` absolute + on-domain
- **R7** — JSON-LD incl. **LocalBusiness/SelfStorage** schema
- **R8** — every `<img>` has descriptive `alt` + explicit `width`/`height`
- **R9** — no duplicate content photo on the same page (logos/icons/process exempt; gallery index exempt)
- **R10** — no broken internal links / no missing referenced images
- **R11** — no `http://` mixed content; no links to `index.html` (clean pretty-URLs)
- **R12** — word-count floors (see table below)
- **R13** — FAQPage schema required on location pages (*warn* elsewhere); visible FAQ set must equal schema FAQ set
- **R14** — phone numbers must be clickable `tel:` links (*warn*)
- **R15** — descriptive link text; no generic anchors ("click here", "read more", "learn more", "here", "more", "find out more"); no empty link text
- **R16** — each image ≤200 KB
- **R17** — `noindex` only on 404 + legacy landing pages (home/, contact/, quote/)
- **R19** — descriptive image filenames (no `IMG_`/`DSC`/numeric; 3–6 keyword words)

### R12 word-count floors (canonical — main-content words, calc UI excluded)
`Blog ≥2000` · `Location ≥1500` · `Pillar (about/pricing/index) ≥1200` · `FAQ ≥1100` ·
`Service ≥1000` · `Helpful-tips ≥800` · `Nav-hub (areas-we-cover/reviews) ≥700`.
**Exempt (no floor):** contact, quote, privacy, terms, gallery, storage-calculator/size-guide UI, 404.
NOTE: the bundled `seo_scan.py` currently encodes location/pillar/service/nav-hub tiers; when blog,
FAQ-page, or helpful-tips page types are added, **extend its `tier()` function** with `Blog ≥2000`
(etc.) so the scanner stays canonical. Blog pages: floor is **2000 words** — flag in `blog-builder`.

## 3. Schema architecture (already best-in-class — keep it intact)
Every page carries JSON-LD; do not regress it. Current state: **LocalBusiness + SelfStorage +
MovingCompany** on all pages; **BreadcrumbList** on nearly all; **FAQPage** on location/service pages;
**Service** on service pages; `aggregateRating` (5.0/478), `geo`, `areaServed`, `openingHours`,
`priceRange`, `sameAs`. New pages must emit the right stack: location → +FAQPage; service → +Service+FAQPage.
2026 note: Google **removed FAQ rich results** and **deprecated HowTo** — keep the markup (still aids
understanding + AI parsing) but don't expect rich snippets. Article/LocalBusiness/BreadcrumbList still matter.
**NAP source of truth = the LocalBusiness JSON-LD in build-site.py** (Wolves Storage Sussex, Ashington
West Sussex RH20; tel 01903 893731; sussexstoragecompany.co.uk; LAPADA + Checkatrade). Keep NAP byte-identical
everywhere (site, schema, GBP, directories) — inconsistency is a top local-trust killer. Never fabricate
reviews/accreditations/facts.

## 4. On-page craft (learned this build — what actually moved rankings)
- **Titles** templated with postcode + value prop, e.g. `Storage in {Town} ({PC1}) | From £15/wk, We Collect` — beats generic `Storage in {Town}`.
- **Metas** unique, ≤145, benefit-led; no two pages share a description.
- **FAQs capture intent**: phrase a question as the real search ("Do you have self-storage units in {Town}?") then answer with the managed-alternative angle.
- **Descriptive anchors only** (R15) — never "click here"; use entity-rich anchors ("container storage in Worthing").
- **Image discipline**: keyword filenames (R19), explicit width/height (R8, prevents CLS), ≤200 KB (R16), real alt. Optional-chaining in Alpine `:src` (`g[i]?.s||''`) avoids `>` breaking scans.
- **Location pages are 52–63% unique** (genuinely not doorway pages) — keep per-town content bespoke; never reuse boilerplate or they collapse into near-duplicates.

## 5. Local SEO levers (biggest off-site opportunity)
- **The single biggest local lever is OFF-site**: Google Business Profile (primary category, services,
  hours, photos), citations/directories, and reviews — NOT on-page blog content. Blog/location pages build
  topical authority that lifts organic; the **map pack** is driven by GBP + reviews + NAP consistency.
- Keep NAP identical across site, GBP, and every directory. Build the entity with `sameAs` (GBP, Companies
  House, LinkedIn, socials). Disambiguate "Ashington, **West Sussex**" (bigger Ashington in Northumberland).
- Brand **mentions** (even unlinked) correlate ~3× backlinks for AI-Overview visibility — pursue local
  press/citations (West Sussex County Times, The Argus, BBC Sussex). See `blog-builder` §digital-PR.

## 6. 2026 algorithm reality (cross-ref [[blog-advanced-craft]])
- Helpful-Content is folded into core ranking — continuous, **site-wide** (one thin page can drag the domain).
- **E-E-A-T**: Trust + first-hand **Experience** are the wedge for a small near-YMYL local business; you
  can't out-authority chains but can out-experience/out-trust them (named owner, real photos, transparent £15/wk).
- **Originality** is rewarded, paraphrase/rehash demoted (Jan-2025 rater guidelines: "Effort, Originality, Value-Add").
- **AI Overviews / GEO**: get cited via answer-first self-contained passages + cited stats/quotes; keyword
  stuffing LOWERS AI visibility. Citations diverge from blue links → small site can be cited without ranking #1.
- **Topical authority** builds faster than domain authority — depth on a focused niche beats broad generalists.

## 7. Safe-edit discipline (hard-won — see the corruption lesson)
- Prefer the **Edit tool** with a unique anchor. For scripted edits on build-site.py use a **guarded
  script**: unique anchor + `assert count == 1` + span-size sanity assert + `assert "def " not in span` +
  `ast.parse()` before write. Never a bare `s.replace(...)` on an ambiguous string.
- The pre-compiled Tailwind means **new utility classes silently fail** — use scoped `<style>` blocks with
  namespaced prefixes for bespoke styling. `fix_amps()` escapes raw `&` (excludes `&&`/entities).
- Recovery net: the harness keeps file history under `~/.claude/file-history/<session>/...`.
- After every edit: rebuild → scan → verify render. Report failures honestly (if scan fails, say so).

## 9. How Google + AI ACTUALLY rank in 2026 (mechanics — full detail in [[seo-2026-foundations]])
Evidence-graded from the Google Content Warehouse leak + DOJ testimony + 2026 studies. These drive every decision above.
- **NavBoost: clicks are a ranking signal** (confirmed). The prize = `lastLongestClick` (did the page END the search?). Sliced by country/device → **design every page to end the search for a West Sussex mobile user** (sizes, price, map, access hours, tap-to-call above fold; no pogo-stick back to SERP). This is the small site's #1 edge over national chains.
- **`siteAuthority` is real** — chains carry it on every page → DON'T fight broad head terms; win local/intent long-tail.
- **Small-specialist advantages** (confirmed signals): `siteFocusScore`/`siteRadius` (stay ruthlessly on the managed-storage niche), `contentEffort`/`OriginalContentScore` (first-hand specifics beat templated boilerplate), and the **site-diversity cap** (chains can't take all SERP slots — be the best independent).
- **`pandaDemotion` is SITE-WIDE** — even 2–3 thin/near-duplicate pages drag the whole small domain ("algorithmic debt"). NEVER spin up templated town pages with swapped names (also = "scaled content abuse" + non-indexing risk). Fewer, deeper, distinct pages win.
- **2026 indexing is selective** — "Crawled, currently not indexed" = a quality verdict; each page must earn its place.
- **2026 updates reward first-party ORIGINATORS, demote aggregators/UGC** — a real West Sussex operator IS a first-party source; lean into first-hand specifics. Avoid self-promo listicles (Jan-2026 crackdown).
- **AI search erodes the chains' moat:** ~2/3 of AI citations come from OUTSIDE the organic top-10, and local intent is the least AI-colonised surface (~7%). Win via self-contained ~130–170-word answer passages + GBP.
- **Static HTML = structural win:** no major AI crawler renders JS except Gemini/Applebot → our site is 100% visible to ChatGPT/Perplexity/Copilot while JS competitors go blank. **Never move content/schema/meta/canonical/NAP into JS.**
- **Authority = brand + entity, not links** — brand mentions correlate ~3× backlinks for AI visibility; branded search feeds siteAuthority + NavBoost. Build the entity (Organization `sameAs` + Wikidata + byte-identical NAP) and manufacture name-demand.

## 9b. Deep mechanics — reason from first principles (full detail in [[seo-2026-deep-mechanics]])
Primary-source (IR papers, patents, DOJ record, RAG papers, Quality Rater Guidelines read in full).
- **3 gates, don't conflate them:** Retrieval (lexical BM25 + dense RankEmbed — win BOTH: exact local tokens AND clean semantics) → Core scoring (Topicality **T\*** = hand-crafted Anchors+Body+Clicks; **Q\*** = static, query-independent, SITE-level trust prior, PageRank-fed) → Re-ranking (NavBoost/Twiddlers on the shortlist) → SERP/Glue (map pack).
- **The math kills folklore:** BM25 TF-saturation = keyword stuffing is mathematically pointless (Nth repeat ≈ worthless); LambdaMART trains on |ΔNDCG| = top positions matter disproportionately by design; E-E-A-T is NOT a score (it's the rater rubric → IS scores → trains the models) → build the machine-observable CORRELATES of trust.
- **NavBoost = the DBN click model:** rewards `lastLongestClick` = the page that ENDS the search (satisfaction s_u). "Squashing" is sub-linear → a chain's 100× volume compresses to ~2-3× → the fight is satisfaction RATIO, not volume. **Cold-start is quantified: ~90-370 distinct-user clicks per geo×device slice over 13 months** for a stable signal — a new page on a small-town query falls below it. WHY LOCAL WINS ANYWAY: geo-slicing strips the chains' national click mountain (their town-slice may be thinner than yours) + squashing caps them → out-SATISFY them on the "[town] container storage" slice.
- **The cold-start ESCAPE (highest strategic priority):** P1 **branded search** (a navigational name-search + long dwell = near-perfect, unfakeable satisfaction vote; builds Q\*/entity; spills to non-branded; **generated OFFLINE** via van livery/signage/flyers/local press → bypasses the rank-to-get-clicks trap; even 10-30/mo seeds it) ≈ P2 **GBP/Glue** (~70% of local clicks in the map pack; directions = "intent to visit") > P3 SERP-CTR (locally-specific titles + rich-result schema lift a_u) > P4 anti-pogo page design (END the search above the fold). The market-recon target queries (container + small-town long-tail) are MECHANICALLY correct: thin-incumbent slices where T\*+proximity rank you without clicks while you bootstrap.
- **Entity = prerequisite for AI citation:** LLM parametric memory is popularity-weighted → a long-tail local brand WON'T be recalled from weights and scaling won't fix it → only the RETRIEVAL path works, and it needs UNAMBIGUOUS entity resolution. Defeat the "Ashington→Northumberland" prior with locally-coherent co-occurrence (West Sussex town + postcode constellation). Byte-consistent NAP + one `@id`/`@graph` + `sameAs` (the disambiguation substrate). Becoming "known" = crossing a fact-confidence threshold via independent corroboration, not "getting listed."
- **Citation = ENTAILMENT** (RAG science): an AI engine cites a passage iff it ENTAILS the model's sentence → atomic, declarative, self-contained claim sentences (one chunk = one topic = geometry; no orphan pronouns; front-load the answer). Fluency correlates NEGATIVELY with grounding → plain factual statements beat polished prose. Use REAL figures (£15/wk, sizes, years, accreditation); never fabricate.
- **Trust is a CEILING (QRG §3.4/§14.0):** Trust is the CENTER of E-E-A-T — an untrustworthy page is capped regardless of relevance/expertise. §5.5: a store with thin identity/customer-service info → rated LOW (so radical transparency — named entity, company no., full address, staffed phone, complaints/claims/terms/insurance — is the highest-ROI trust fix). §4.6.5: thin templated near-duplicate town pages = Lowest "no matter how created", judged SITE-wide → never build them. Storage = near-YMYL (financial + custody of possessions).
