# Blog Builder — Wolves Storage Sussex

Operational playbook for building blog/guide pages that are a clear step above every competitor.
Built from research banked in memory: `blog-competitor-research` (round 1) + `blog-advanced-craft` (round 2).
Full citable storage facts: see `storage-science.md` in this skill folder.

**The north star:** Google's "good content" and AI's "citable content" have converged. Both reward
clear, answer-first, evidence-dense writing backed by first-hand experience on a focused niche.
One pattern serves blue links, featured snippets, and AI Overviews. Out-EXPERT competitors
(domain science + your own data + first-hand experience) — do not out-publish them.

---

## 0. Before anything: read the SERP (the gate)
Run the target query in an incognito Google set to the service area FIRST.
- **Local pack (map 3-pack) showing?** → an article CANNOT win this. It needs the GBP + a
  service/location page, NOT a blog post. Stop; pick a different (informational) query.
- **Featured snippet is a table?** → your page needs a real `<table>`. List? → list. Match the format.
- **Reddit/forum results ranking?** → lean hard into first-hand operator experience.
- **Profile the ranking PAGES** (their depth/backlinks), not the keyword. Target queries where
  page-1 is thin/weak/stale, or where AI Overviews cite non-top-10 pages (small-site opening).
- **One URL per dominant intent.** Informational → blog. Commercial/local → service page. Never both.

## 1. Originality hard gate (do not publish without passing)
Name ≥1 element in the draft — a stat, an original photo, a process detail, a tested result, a
named first-hand story — that appears NOWHERE on page 1 of Google for this query.
If you can't, the post is a rehash. Get the missing element first. Sources of original gain you own:
- Proprietary booking data ("What West Sussex puts in storage", seasonality, avg duration, containers/property)
- First-hand process + failure modes (real damp/mould/woodworm you've seen on collections)
- Original photos/diagrams of the real collect-pack-seal-redeliver operation
- Real customer-call objections
"Longer than rivals" is dead — rehash is rated Lowest by Google's 2026 rater guidelines.

## 2. The ONE writing pattern (every key section)
```
<h2>[the exact question a user searches]</h2>
<p>[40–60 word, PRONOUN-FREE, NAMED-SUBJECT declarative answer — first sentence stands alone]</p>
<p>[then 100–400 words: a real STAT with a cited source, and/or a tight HTML comparison table]</p>
```
- Lead answer wins the featured snippet AND is the perfect AI-retrieval chunk; the stat+citation wins the AI citation.
- **No orphan pronouns** (most-violated rule): never "It costs…/They collect…". Always "Managed container storage costs…".
  Test: copy any paragraph into a blank doc — does it answer with zero reliance on the paragraph above? If not, rewrite.
- Snippet ceilings (measured): paragraph ≤320 chars / 40–55 words; list ≤8 items, ~3 words each
  (Google often builds the list from H2/H3 subheadings — so make steps real headings); table ≤5 rows × 3 cols,
  cells ≤3 words, MUST be a real `<table>` (CSS/div tables are not lifted).
- AI-citation levers (Princeton GEO): cite reputable sources (+115%), add statistics (+37–41%),
  add quotations (+28–32%). NEVER keyword-stuff (−10%, the only tactic that lowers AI visibility). Never invent stats.

## 3. Entity / topical-authority rules
- One URL = one entity, named consistently ("managed container storage" — don't drift to "lock-up"/"unit" mid-page).
- Open the page with a definition sentence (raises entity salience).
- Always disambiguate the place: "Ashington, **West Sussex**" (there's a bigger Ashington in Northumberland).
- Everything must connect back to the niche (managed storage for a West Sussex customer). No off-topic filler
  (no generic "best moving boxes" listicles) — unaligned traffic dilutes topical authority.
- Internal links: BODY links > nav/footer. Put an entity-rich, descriptive anchor EARLY in the prose.
  Every blog post links UP to its relevant service/money page with a descriptive anchor (never "click here").

## 4. E-E-A-T + conversion (built into every post)
Wedge = TRUST + EXPERIENCE (can't out-authority chains; can out-experience/out-trust them).
- **Byline** to the named owner → links to a living bio/entity page (with `sameAs` to Checkatrade/LAPADA/GBP).
- **≥2 first-hand markers** ("we've collected sofas across Ashington/Worthing/Storrington for X years…").
- **≥1 original photo/video** of the real operation (never stock).
- **Transparent £15/week + terms** stated or linked (trust signal AND differentiator vs quote-gated rivals).
- **Credentials linked OUT and explained** (3rd-party corroboration beats self-claims).
- Fix the universal competitor flaw "capture-but-don't-convert": one early in-body link to the service page →
  mid-article soft CTA after a value moment → end-of-post TRUST-WRAPPED hard CTA → a "next stage" link.
  Trust-wrapped CTA block = headline + 3-step process + "£15/week · no hidden fees" + live Checkatrade rating
  + "family-run, Ashington" + first-person button ("Book my collection") + "free quote · no obligation" beside it.

## 5. Domain expertise = the un-fakeable edge
Competitors are generic because they lack real storage science. Use `storage-science.md` (this folder) for
cited facts. The 5 unique angles no competitor touches:
1. The dew-point reason unheated metal/drive-up units "sweat" in a UK winter.
2. The 70% RH mould clock (mould in ~3 months at 70%, days at 90%) → storage duration is decisive.
3. Don't seal wood/leather/upholstery in airtight plastic (the science competitors get wrong).
4. Damp/mould is the loss insurance won't pay (standard exclusion) → the dry environment is the real risk control.
5. Managed packing likely creates a BAILMENT = a higher legal duty of care than a drive-up "own-risk" unit.
HONESTY GUARDRAIL: position the warehouse as clean/dry/ventilated/managed (avoids the cold-metal condensation trap)
— do NOT imply museum-grade climate control unless RH/temp are actually monitored on site. Never invent figures.

## 6. Build mechanics (this site specifically)
- Pages are generated by `build-site.py`. Tailwind is PRE-COMPILED → new utility classes silently fail.
  Use a scoped `<style>` block with a namespaced prefix (e.g. `.blg-`) for any bespoke blog styling.
- Alpine.js is available globally; `fix_amps()` escapes raw `&`. Match the existing page idiom.
- Schema per post: **Article + BreadcrumbList + LocalBusiness** (+ FAQ-style Q&A text). Note: Google REMOVED
  FAQ rich results (2026) and deprecated HowTo — keep FAQ markup/text (still aids understanding + AI parsing),
  but don't expect rich snippets. Article + LocalBusiness + BreadcrumbList still matter.
- Mandatory site rules (from memory): keep SEO rules (R1–R19 + word floors) PASS; follow the design language of
  the service-page template; AUTO-ADD every new page to `sitemap.xml` AND root `llms.txt`. Never change the site
  without asking the user first.
- **Also apply the `seo` skill** — every blog page must pass the full SEO bible. Run its bundled
  `.claude/skills/seo/seo_scan.py` + `bug_scan.py` after build (target: 0 hard issues). **R12 blog word
  floor = 2000 words** — extend the scanner's `tier()` with a blog tier when the first blog page is built.

## 7. Anti-patterns — never ship these (seen in competitor blogs)
- Thin <700-word generic listicles ("5 things", "10 reasons").
- Title–intent mismatch (promise prices/examples then withhold — every chain does this; we don't).
- Keyword cannibalisation (multiple near-duplicate pages for one topic → one canonical page per topic).
- A whole blog dumped in one AI batch (stales together, cannibalises).
- Storage bolted on as an afterthought / CTA pointing to the homepage instead of the service page.
- Off-topic seasonal "things to do" filler; PDF-trapped guides; stale never-refreshed evergreens.
- Uncited/random-source stats. Use real authorities (SSA UK, PD5454/National Archives, CCI, FCA, ONS, gov.uk).

## 8. Final QA checklist (every post, before publish)
- [ ] SERP read done; this is genuinely an informational query a blog post can win (§0)
- [ ] Originality gate passed — ≥1 element found nowhere on page 1 (§1)
- [ ] Each key section: question H2 → 40–60w named-subject answer → stat/table (§2); no orphan pronouns
- [ ] Definition-first; "Ashington, West Sussex"; consistent entity naming (§3)
- [ ] ≥1 early in-body link UP to the service page with a descriptive anchor (§3/§4)
- [ ] Owner byline → bio page; ≥2 first-hand markers; ≥1 original photo; £15/wk stated (§4)
- [ ] End-of-post trust-wrapped CTA + a "next stage" link (§4)
- [ ] Domain-science facts cited from storage-science.md where relevant; honesty guardrail respected (§5)
- [ ] Scoped `.blg-` styling; Article+BreadcrumbList+LocalBusiness schema; FAQ text (§6)
- [ ] SEO rules PASS; word floor met; auto-added to sitemap.xml + llms.txt (§6)
- [ ] No anti-patterns (§7); real cited sources only
- [ ] Build runs clean; verified rendered output

## Build order (when the user greenlights a blog)
HUB FIRST: evergreen core guides (size/what-fits, cost/£15-week, packing, what-can't-you-store, moving checklist),
each with the §2 pattern + §4 CTA + schema. THEN the outer informational cluster (managed-vs-self, damp/mould,
downsizing/between-homes, local "moving to [town]" guides). THEN 3–4 linkable assets (a "Sussex Storage Cost Index"
from real booking data; a dual-purpose calculator that gives a private estimate AND publishes a public benchmark)
+ a 1hr/week local-PR habit. A hub pays off immediately; a rolling blog only pays at a cadence you can sustain.
